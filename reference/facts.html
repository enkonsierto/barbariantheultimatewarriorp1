Programmer's hidden signature
At 84D0, between the music engine routines, there are four bytes with ASCII values 61 64 61 6D -- the word ADAM. This code is completely unreachable at runtime; no code path can ever reach it. No member of the development team is known as Adam, and at the time of disassembly, the reason for including it as a signature in the game was unknown. Perhaps a programmer's nickname? A family member's name? Perhaps a reference to Masters of the Universe?
A jump that never jumps
At melody_running, inside the music engine, there is a JR Z,84D4 instruction -- a conditional branch that would jump to itself. However, the Z flag is always clear when execution reaches that point, so the branch never fires. It is dead code that does nothing.
The winner's exit animation that never plays
Both p1_wins_seq (P1 wins) and p2_wins_seq (P2 wins) contain unreachable blocks that would have applied the animation 35 (walk-out animation). However it is completely unreachable: they execute a JP or JR immediately beforehand and never return. In other systems this animation is executed, showing the winner walking out of the arena.
The game patches itself to enable damage
A RET Z instruction inside p1_energy_gate prevents P1 from losing energy, keeping them invulnerable during the demo/attract mode. When the player starts a real match, routines one_player_mode and two_player_mode overwrite that byte in memory with a NOP, removing the protection. The same pair of routines also rewrites the menu text from "0 DEMO" to "START GAME". Once this routines are executed we can't no longer launch the demo mode.
Music loop pointers that are never written
The music engine state area at melody_note reserves two 2-byte slots for loop-back addresses: Melody loop pointer at melody_loop_ptr and Bass loop pointer at bass_loop_ptr. In the final version neither is ever written to. The restart mechanism instead reads the loop address inline from the music data stream at restart_music. The slots are a leftover from an earlier design.
Unused final boss
The enemy and head sprite bank at other_sprites contains the complete tiles of Drax and Mariana with no runtime caller. No sprite frame definition references these indices and no movement applies them during gameplay. At B717 there are 16 bytes of tile sequence data corresponding to the spell projectile animation for Drax, alternating tile IDs 18 and 22 from the head sprite bank. No runtime code references this data. They are remnants of the final boss cut in this part.
An unused buffer initialised to 6
At unused_buffer there are 2 bytes. The first is always 0, written redundantly by decap_setup. The second, at B6EA retains its initialisation value of 6 and is neither read nor written at any point. It is likely a remnant from when the P2 overlay state block had 9 fields and the IY+8 slot was later removed.
Font data overwritten by the framebuffer
From 6770 onwards there are UDG font table entries that were loaded as part of the main code block but are never referenced at runtime. The framebuffer, which spans framebuffer to 687F overwrites them during normal gameplay.
The interrupt vector table overwrites the first byte of tile 0
The interrupt vector table generator at i_v_t_generator writes 257 bytes of value 0xF0 starting from address BF00. The 257th byte lands exactly at C000 the first byte of the barbarian sprite tileset. After startup that byte holds F0 rather than the original 00. Because tile 0 is the fully transparent blank tile and its header byte is not read by any rendering path once the game is running, the change has no visible effect.
The X-delta table and the ISR overwrites two unused sprite tiles
At startup, BF27 copies 100 bytes from x_delta_table (the per-frame horizontal displacement table) to address F080. This area corresponds to tiles 194 and 195 in the barbarian sprite bank. In the other hand, BF87 writes the 46-byte ISR code -- relocated from int_serv_routine -- to address F0F0. This address falls inside tiles 195 and 196 of the barbarian sprite bank (F0C0-F0FF). Those tiles are unused: no sprite definition references either index.
P2 is rendered by compositing one overlay tile on top of the shared P1 sprite
P1 and P2 share an identical base sprite bank at barb_sprite_tileset. Rather than maintaining a full duplicate tileset for P2, the game renders P2 using the same 4x3 tile grid as P1 and then composites a single extra tile on top each frame via p2_overlay_tile. The overlay tile index is read from p2_overlay_tile_table, and its framebuffer position from p2_overlay_pos_table, both indexed by the current P2 frame counter at p2_frame_index. The tile is sourced from the secondary sprite bank at other_sprites, which holds the P2-specific head variants and overlay graphics. To avoid duplicating renderer logic, p2_overlay_tile populates the overlay state block at p2_overlay_block_state using the same 9-byte layout as the flying head block at head_state, allowing render_head_tile and head_mirroring to serve both paths without modification.
A hidden build stamp is accessible from the main menu
Pressing key 3 on the main menu does not launch a game mode. Instead, main_menu loads the address BE16 into DE and draws the string "FINAL V8 5>11>87" over the SELECT OPTION line via draw_word, then loops straight back to the animation and key-wait cycle. The string remains on screen until the player presses another key. This is is consistent with a build tag left by the developer: version 8, dated 5 November 1987.
